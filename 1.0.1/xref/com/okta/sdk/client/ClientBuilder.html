17  package com.okta.sdk.client;
18  
19  import com.okta.sdk.authc.credentials.ClientCredentials;
20  import com.okta.sdk.cache.CacheManager;
21  
22  /**
23   * A <a href="http://en.wikipedia.org/wiki/Builder_pattern">Builder design pattern</a> used to
24   * construct {@link com.okta.sdk.client.Client} instances.
25   *
26   * <p>The {@code ClientBuilder} is used to construct Client instances with Okta credentials,
27   * Proxy and Cache configuration.  Understanding caching is extremely important when creating a Client instance, so
28   * please ensure you read the <em>Caching</em> section below.</p>
29   *
30   * <h1>Usage</h1>
31   *
32   * <p>The simplest usage is to just call the {@link #build() build()} method, for example:</p>
33   *
34   * <pre>
35   * Client client = {@link Clients Clients}.builder().{@link #build() build()};
36   * </pre>
37   *
38   * <p>This will:</p>
39   * <ul>
40   *   <li>Automatically enable a simple in-memory {@link CacheManager} for enhanced performance (but please read
41   *       the <em>Caching</em> section below for effects/warnings).</li>
42   *   <li>Automatically attempt to find your API credentials values in a number of default/conventional locations and then use
43   *       the discovered values. Without any other configuration, the following locations will be each be checked,
44   *       in order:</li>
45   * </ul>
46   *
47   * <ol>
48   *     <li>The environment variable {@code OKTA_CLIENT_TOKEN}.  If either of
49   *         these values are present, they override any previously discovered value.</li>
50   *     <li>A yaml file that exists at the file path or URL specified by the {@code okta.client.file}
51   *         system property.  If this file exists and any values are present, the values override any
52   *         previously discovered value.  The {@code okta.client.file} system property String can be an
53   *         absolute file path, or it can be a URL or a classpath value by using the {@code url:} or
54   *         {@code classpath:} prefixes respectively. Default value is {code ~/.okta/okta.yaml}. </li>
55   *     <li>The system properties {@code okta.client.token}.  If this value is present, it will override any
56   *     previously discovered values.</li>
57   * </ol>
58   *
59   * <p><b>SECURITY NOTICE:</b> While the {@code okta.client.token} system property may be used to represent your
60   * API Key Secret as mentioned above, this is not recommended: process listings on a machine will expose process
61   * arguments (like system properties) and expose the secret value to anyone that can read process listings.  As
62   * always, secret values should never be exposed to anyone other than the person that owns the API Key.</p>
63   *
64   * <p>While an API Key ID may be configured anywhere (and be visible by anyone), it is recommended to use a private
65   * read-only file or an environment variable to represent API Key secrets.  <b>Never</b> commit secrets to source code
66   * or version control.</p>
67   *
68   * <h2>Explicit API Key Configuration</h2>
69   *
70   * <p>The above default API Key searching heuristics may not be suitable to your needs.  In that case, you will likely
71   * need to explicitly configure your API Key.  For example:</p>
72   *
73   * <pre>
74   * ClientCredentials clientCredentials = new TokenClientCredentials("apiToken");
75   *
76   * Client client = {@link Clients Clients}.builder().setClientCredentials(clientCredentials).build();
77   * </pre>
78   * *
79   * <h1>Caching</h1>
80   *
81   * NOTE: Caching support is currently in Alpha status.
82   *
83   * <p>By default, a simple production-grade in-memory {@code CacheManager} will be enabled when the Client instance is
84   * created.  This {@code CacheManager} implementation has the following characteristics:</p>
85   *
86   * <ul>
87   *     <li>It assumes a default time-to-live and time-to-idle of 1 hour for all cache entries.</li>
88   *     <li>It auto-sizes itself based on your application's memory usage.  It will not cause OutOfMemoryExceptions.
89   *         (It does this by retaining only 100 strong references to cached objects.  Additional cached objects are
90   *          weakly referenced, ensuring the garbage collector can evict weakly referenced cache entries if it needs
91   *          more memory for your application.).
92   *     </li>
93   * </ul>
94   *
95   * <p>but, please note:</p>
96   *
97   * <p><b>The default cache manager is not suitable for an application deployed across multiple JVMs.</b></p>
98   *
99   * <p>This is because the default implementation is 100% in-memory (in-process) in the current JVM.  If more than one
100  * JVM is deployed with the same application codebase - for example, a web application deployed on multiple identical
101  * hosts for scaling or high availability - each JVM would have it's own in-memory cache.  Multiple disconnected caches
102  * for the same data will cause cache coherency problems and likely cause errors in your application!</p>
103  *
104  * <p>As a result, if your application that uses a Okta Client instance is deployed across multiple JVMs, you
105  * SHOULD ensure that the Client is configured with a {@code CacheManager} implementation that uses coherent and
106  * clustered/distributed memory.</p>
107  *
108  * <h2>Custom CacheManager</h2>
109  *
110  * <p>If you want to specify a custom {@code CacheManager} implementation:</p>
111  *
112  * <pre>
113  * CacheManager cacheManager = new MyCacheManagerImplementation();
114  * Client client = {@link com.okta.sdk.client.Clients Clients}.builder().<b>setCacheManager(cacheManager)</b>.build();
115  * </pre>
116  *
117  * <h3>Application deployed on a single JVM</h3>
118  *
119  * <p>If your application is deployed on a <b>single JVM</b> and you still want to use the default
120  * {@code CacheManager} implementation, but the default cache configuration does not meet your
121  * needs, you can specify a different configuration.  For example:</p>
122  *
123  * <pre>
124  * import static com.okta.sdk.cache.Caches.*;
125  *
126  * ...
127  *
128  * {@link com.okta.sdk.cache.Caches Caches}.{@link com.okta.sdk.cache.Caches#newCacheManager() newCacheManager()}
129  *     .withDefaultTimeToLive(300, TimeUnit.SECONDS) // default
130  *     .withDefaultTimeToIdle(300, TimeUnit.SECONDS) //general default
131  *     .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(Account.class) //Account-specific cache settings
132  *         .withTimeToLive(1, TimeUnit.HOURS)
133  *         .withTimeToIdle(30, TimeUnit.MINUTES))
134  *     .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(Group.class) //Group-specific cache settings
135  *         .withTimeToLive(2, TimeUnit.HOURS))
136  *
137  *     //... etc ...
138  *
139  *     .build(); //build the CacheManager
140  * </pre>
141  *
142  * <p>See the {@link com.okta.sdk.cache.Caches Caches} utility class and the
143  * {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder} docs for more information.</p>
144  *
145  * <h3>Application deployed across multiple JVMs</h3>
146  *
147  * <p>If your application is deployed across multiple JVMs (for example a web app deployed on multiple web nodes for
148  * scale and/or high availability), you will likely need to specify a custom
149  * {@code CacheManager} implementation that is based on network distributed/coherent memory.  For example, an
150  * implementation might delegate to a <a href="http://hazelcast.org/">Hazelcast</a> or
151  * <a href="http://redis.io/">Redis</a> cluster. For example, if using the out-of-the-box Hazelcast plugin:</p>
152  *
153  * <pre>
154  * import com.okta.sdk.hazelcast.HazelcastCacheManager;
155  * // ... etc ...
156  *
157  * //Get a HazelcastInstance from your app/config.  This can be a HazelcastClient instance too:
158  * HazelcastInstance hazelcastInstance = getHazelcastInstanceOrHazelcastClient();
159  *
160  * CacheManager cacheManager = new HazelcastCacheManager(hazelcastInstance);
161  * Client client = {@link com.okta.sdk.client.Clients Clients}.builder().setCacheManager(cacheManager).build();
162  * </pre>
163  *
164  * <p><b>NOTE</b>: it should be noted that <a href="http://memcached.org/">Memcache</a> <em>DOES NOT</em> guarantee cache
165  * coherency.  It is strongly recommended that you do not use Memcache as your clustered caching solution (memcache
166  * is fine for caching files, etc, but not data that is expected to be coherent across multiple cluster nodes).</p>
167  *
168  * <h4>Disable Caching</h4>
169  *
170  * <p>While production applications will usually enable a working CacheManager as described above, you might wish to disable caching
171  * entirely when testing or debugging to remove 'moving parts' for better clarity into request/response
172  * behavior.  You can do this by configuring a <em>disabled</em> {@code CacheManager} instance. For example:</p>
173  *
174  * <pre>
175  * Client client = {@link com.okta.sdk.client.Clients Clients}.builder().setCacheManager(
176  *     <b>{@code Caches.newDisabledCacheManager()}</b>
177  * ).build();
178  * </pre>
179  *
180  * <h1>Single Instance</h1>
181  *
182  * <p>Finally, it should be noted that, after building a {@code client} instance, that same instance should be used
183  * everywhere in your application. Creating multiple client instances in a single application could have
184  * negative side effects:</p>
185  *
186  * <p>As mentioned above, a client has a {@link #setCacheManager(com.okta.sdk.cache.CacheManager) CacheManager}
187  * reference.  If your application uses multiple client instances, each client's referenced CacheManager would likely
188  * become out of sync with the others, making your cache
189  * <a href="http://en.wikipedia.org/wiki/Cache_coherence">incoherent</a>.  This will likely
190  * result in exposing stale data to your application and could data errors.</p>
191  *
192  * <p>If you must have multiple {@code Client} instances in your application, you should ensure that each client
193  * references the same exact {@code CacheManager} instance to guarantee cache coherency.</p>
194  *
195  * @since 0.5.0
196  */
197 public interface ClientBuilder {
198 
199     String DEFAULT_CLIENT_API_TOKEN_PROPERTY_NAME = "okta.client.token";
200     String DEFAULT_CLIENT_CACHE_ENABLED_PROPERTY_NAME = "okta.client.cache.enabled";
201     String DEFAULT_CLIENT_CACHE_TTL_PROPERTY_NAME = "okta.client.cache.defaultTtl";
202     String DEFAULT_CLIENT_CACHE_TTI_PROPERTY_NAME = "okta.client.cache.defaultTti";
203     String DEFAULT_CLIENT_CACHE_CACHES_PROPERTY_NAME = "okta.client.cache.caches";
204     String DEFAULT_CLIENT_ORG_URL_PROPERTY_NAME = "okta.client.orgUrl";
205     String DEFAULT_CLIENT_CONNECTION_TIMEOUT_PROPERTY_NAME = "okta.client.connectionTimeout";
206     String DEFAULT_CLIENT_AUTHENTICATION_SCHEME_PROPERTY_NAME = "okta.client.authenticationScheme";
207     String DEFAULT_CLIENT_PROXY_PORT_PROPERTY_NAME = "okta.client.proxy.port";
208     String DEFAULT_CLIENT_PROXY_HOST_PROPERTY_NAME = "okta.client.proxy.host";
209     String DEFAULT_CLIENT_PROXY_USERNAME_PROPERTY_NAME = "okta.client.proxy.username";
210     String DEFAULT_CLIENT_PROXY_PASSWORD_PROPERTY_NAME = "okta.client.proxy.password";
211 
212     /**
213      * Allows specifying an {@code ApiKey} instance directly instead of relying on the
214      * default location + override/fallback behavior defined in the {@link ClientBuilder documentation above}.
215      *
216      * Currently you should use a com.okta.sdk.impl.api.TokenClientCredentials (if you are NOT using an okta.yaml file)
217      *
218      * @param clientCredentials the token to use to authenticate requests to the Okta API server.
219      * @return the ClientBuilder instance for method chaining.
220      */
221     ClientBuilder setClientCredentials(ClientCredentials clientCredentials);
222 
223     /**
224      * Sets the HTTP proxy to be used when communicating with the Okta API server.  For example:
225      *
226      * <pre>
227      * Proxy proxy = new Proxy("whatever.domain.com", 443);
228      * Client client = {@link Clients Clients}.builder().setProxy(proxy).build();
229      * </pre>
230      *
231      * @param proxy the {@code Proxy} you need to use.
232      * @return the ClientBuilder instance for method chaining.
233      */
234     ClientBuilder setProxy(Proxy proxy);
235 
236     /**
237      * Sets the {@link CacheManager} that should be used to cache Okta REST resources, reducing round-trips to the
238      * Okta API server and enhancing application performance.
239      *
240      * <h1>Single JVM Applications</h1>
241      *
242      * <p>If your application runs on a single JVM-based applications, the
243      * {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder} should be sufficient for your needs. You
244      * create a {@code CacheManagerBuilder} by using the {@link com.okta.sdk.cache.Caches Caches} utility class,
245      * for example:</p>
246      *
247      * <pre>
248      * import static com.okta.sdk.cache.Caches.*;
249      *
250      * ...
251      *
252      * Client client = Clients.builder()...
253      *     .setCacheManager(
254      *         {@link com.okta.sdk.cache.Caches#newCacheManager() newCacheManager()}
255      *         .withDefaultTimeToLive(1, TimeUnit.DAYS) //general default
256      *         .withDefaultTimeToIdle(2, TimeUnit.HOURS) //general default
257      *         .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(Account.class) //Account-specific cache settings
258      *             .withTimeToLive(1, TimeUnit.HOURS)
259      *             .withTimeToIdle(30, TimeUnit.MINUTES))
260      *         .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(Group.class) //Group-specific cache settings
261      *             .withTimeToLive(2, TimeUnit.HOURS))
262      *         .build() //build the CacheManager
263      *     )
264      *     .build(); //build the Client
265      * </pre>
266      *
267      * <p><em>The above TTL and TTI times are just examples showing API usage - the times themselves are not
268      * recommendations.  Choose TTL and TTI times based on your application requirements.</em></p>
269      *
270      * <h1>Multi-JVM / Clustered Applications</h1>
271      *
272      * <p>The default {@code CacheManager} instances returned by the
273      * {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder} might not be sufficient for a
274      * multi-instance application that runs on multiple JVMs and/or hosts/servers, as there could be cache-coherency
275      * problems across the JVMs.  See the {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder}
276      * JavaDoc for additional information.</p>
277      *
278      * <p>In these multi-JVM environments, you will likely want to create a simple CacheManager implementation that
279      * wraps your distributed Caching API/product of choice and then plug that implementation in to the Okta SDK
280      * via this method.  Hazelcast is one known cluster-safe caching product, and the Okta SDK has out-of-the-box
281      * support for this as an extension module.  See the top-level class JavaDoc for a Hazelcast configuration
282      * example.</p>
283      *
284      * @param cacheManager the {@link CacheManager} that should be used to cache Okta REST resources, reducing
285      *                     round-trips to the Okta API server and enhancing application performance.
286      * @return the ClientBuilder instance for method chaining
287      */
288     ClientBuilder setCacheManager(CacheManager cacheManager);
289 
290     /**
291      * Overrides the default (very secure)
292      * <a href="https://developer.okta.com/docs/api/getting_started/design_principles#authentication">Okta SSWS Digest
293      * Authentication Scheme</a> used to authenticate every request sent to the Okta API server.
294      *
295      * <p>It is not recommended that you override this setting <em>unless</em> your application is deployed in an
296      * environment that - outside of your application's control - manipulates request headers on outgoing HTTP requests.
297      * Google App Engine is one such environment, for example.</p>
298      *
299      * <pre>
300      * Client client = Clients.builder()...
301      *    // setApiKey, etc...
302      *    .setAuthenticationScheme(AuthenticationScheme.SSWS) //set the SSWS authentication scheme
303      *    .build(); //build the Client
304      * </pre>
305      *
306      * @param authenticationScheme the type of authentication to be used for communication with the Okta API server.
307      * @return the ClientBuilder instance for method chaining
308      */
309     ClientBuilder setAuthenticationScheme(AuthenticationScheme authenticationScheme);
310 
311     /**
312      * Sets both the timeout until a connection is established and the socket timeout (i.e. a maximum period of inactivity
313      * between two consecutive data packets).  A timeout value of zero is interpreted as an infinite timeout.
314      *
315      * @param timeout connection and socket timeout in milliseconds
316      * @return the ClientBuilder instance for method chaining
317      */
318     ClientBuilder setConnectionTimeout(int timeout);
319 
320     /**
321      * Sets the base URL of the Okta REST API to use.  If unspecified, this value defaults to
322      * {@code https://api.okta.com/v1} - the most common use case for Okta's public SaaS cloud.
323      *
324      * <p>Customers using Okta's Enterprise HA cloud might need to configure this to be
325      * {@code https://enterprise.okta.io/v1} for example.</p>
326      *
327      * @param baseUrl the base URL of the Okta REST API to use.
328      * @return the ClientBuilder instance for method chaining
329      */
330     ClientBuilder setOrgUrl(String baseUrl);
331 
332     /**
333      * Constructs a new {@link Client} instance based on the ClientBuilder's current configuration state.
334      *
335      * @return a new {@link Client} instance based on the ClientBuilder's current configuration state.
336      */
337     Client build();
338 }
