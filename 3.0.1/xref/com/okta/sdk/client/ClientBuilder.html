28  /**
29   * A <a href="http://en.wikipedia.org/wiki/Builder_pattern">Builder design pattern</a> used to
30   * construct {@link com.okta.sdk.client.Client} instances.
31   *
32   * <p>The {@code ClientBuilder} is used to construct Client instances with Okta credentials,
33   * Proxy and Cache configuration.  Understanding caching is extremely important when creating a Client instance, so
34   * please ensure you read the <em>Caching</em> section below.</p>
35   *
36   * <h1>Usage</h1>
37   *
38   * <p>The simplest usage is to just call the {@link #build() build()} method, for example:</p>
39   *
40   * <pre>
41   * Client client = {@link Clients Clients}.builder().{@link #build() build()};
42   * </pre>
43   *
44   * <p>This will:</p>
45   * <ul>
46   *   <li>Automatically enable a simple in-memory {@link CacheManager} for enhanced performance (but please read
47   *       the <em>Caching</em> section below for effects/warnings).</li>
48   *   <li>Automatically attempt to find your API credentials values in a number of default/conventional locations and then use
49   *       the discovered values. Without any other configuration, the following locations will be each be checked,
50   *       in order:</li>
51   * </ul>
52   *
53   * <ol>
54   *     <li>The environment variable {@code OKTA_CLIENT_TOKEN}.  If either of
55   *         these values are present, they override any previously discovered value.</li>
56   *     <li>The system properties {@code okta.client.token}.  If this value is present, it will override any
57   *     previously discovered values.</li>
58   * </ol>
59   *
60   * <p><b>SECURITY NOTICE:</b> While the {@code okta.client.token} system property may be used to represent your
61   * API Key Secret as mentioned above, this is not recommended: process listings on a machine will expose process
62   * arguments (like system properties) and expose the secret value to anyone that can read process listings.  As
63   * always, secret values should never be exposed to anyone other than the person that owns the API Key.</p>
64   *
65   * <p>While an API Key ID may be configured anywhere (and be visible by anyone), it is recommended to use a private
66   * read-only file or an environment variable to represent API Key secrets.  <b>Never</b> commit secrets to source code
67   * or version control.</p>
68   *
69   * <h2>Explicit API Key Configuration</h2>
70   *
71   * <p>The above default API Key searching heuristics may not be suitable to your needs.  In that case, you will likely
72   * need to explicitly configure your API Key.  For example:</p>
73   *
74   * <pre>
75   * ClientCredentials clientCredentials = new TokenClientCredentials("apiToken");
76   *
77   * Client client = {@link Clients Clients}.builder().setClientCredentials(clientCredentials).build();
78   * </pre>
79   *
80   * <h1>Caching</h1>
81   *
82   * <p>By default, a simple production-grade in-memory {@code CacheManager} will be enabled when the Client instance is
83   * created.  This {@code CacheManager} implementation has the following characteristics:</p>
84   *
85   * <ul>
86   *     <li>It assumes a default time-to-live and time-to-idle of 1 hour for all cache entries.</li>
87   *     <li>It auto-sizes itself based on your application's memory usage.  It will not cause OutOfMemoryExceptions.
88   *         (It does this by retaining only 100 strong references to cached objects.  Additional cached objects are
89   *          weakly referenced, ensuring the garbage collector can evict weakly referenced cache entries if it needs
90   *          more memory for your application.).
91   *     </li>
92   * </ul>
93   *
94   * <p>but, please note:</p>
95   *
96   * <p><b>The default cache manager is not suitable for an application deployed across multiple JVMs.</b></p>
97   *
98   * <p>This is because the default implementation is 100% in-memory (in-process) in the current JVM.  If more than one
99   * JVM is deployed with the same application codebase - for example, a web application deployed on multiple identical
100  * hosts for scaling or high availability - each JVM would have it's own in-memory cache.  Multiple disconnected caches
101  * for the same data will cause cache coherency problems and likely cause errors in your application!</p>
102  *
103  * <p>As a result, if your application that uses a Okta Client instance is deployed across multiple JVMs, you
104  * SHOULD ensure that the Client is configured with a {@code CacheManager} implementation that uses coherent and
105  * clustered/distributed memory.</p>
106  *
107  * <h2>Custom CacheManager</h2>
108  *
109  * <p>If you want to specify a custom {@code CacheManager} implementation:</p>
110  *
111  * <pre>
112  * CacheManager cacheManager = new MyCacheManagerImplementation();
113  * Client client = {@link com.okta.sdk.client.Clients Clients}.builder().<b>setCacheManager(cacheManager)</b>.build();
114  * </pre>
115  *
116  * <h3>Application deployed on a single JVM</h3>
117  *
118  * <p>If your application is deployed on a <b>single JVM</b> and you still want to use the default
119  * {@code CacheManager} implementation, but the default cache configuration does not meet your
120  * needs, you can specify a different configuration.  For example:</p>
121  *
122  * <pre>
123  * import static com.okta.sdk.cache.Caches.*;
124  *
125  * ...
126  *
127  * {@link com.okta.sdk.cache.Caches Caches}.{@link com.okta.sdk.cache.Caches#newCacheManager() newCacheManager()}
128  *     .withDefaultTimeToLive(300, TimeUnit.SECONDS) // default
129  *     .withDefaultTimeToIdle(300, TimeUnit.SECONDS) //general default
130  *     .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(User.class) //User-specific cache settings
131  *         .withTimeToLive(1, TimeUnit.HOURS)
132  *         .withTimeToIdle(30, TimeUnit.MINUTES))
133  *     .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(Group.class) //Group-specific cache settings
134  *         .withTimeToLive(2, TimeUnit.HOURS))
135  *
136  *     //... etc ...
137  *
138  *     .build(); //build the CacheManager
139  * </pre>
140  *
141  * <p>See the {@link com.okta.sdk.cache.Caches Caches} utility class and the
142  * {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder} docs for more information.</p>
143  *
144  * <h3>Application deployed across multiple JVMs</h3>
145  *
146  * <p>If your application is deployed across multiple JVMs (for example a web app deployed on multiple web nodes for
147  * scale and/or high availability), you will likely need to specify a custom
148  * {@code CacheManager} implementation that is based on network distributed/coherent memory.  For example, an
149  * implementation might delegate to a <a href="http://hazelcast.org/">Hazelcast</a> or
150  * <a href="http://redis.io/">Redis</a> cluster. For example, if using the out-of-the-box Hazelcast plugin:</p>
151  *
152  * <pre>
153  * import com.okta.sdk.hazelcast.HazelcastCacheManager;
154  * // ... etc ...
155  *
156  * //Get a HazelcastInstance from your app/config.  This can be a HazelcastClient instance too:
157  * HazelcastInstance hazelcastInstance = getHazelcastInstanceOrHazelcastClient();
158  *
159  * CacheManager cacheManager = new HazelcastCacheManager(hazelcastInstance);
160  * Client client = {@link com.okta.sdk.client.Clients Clients}.builder().setCacheManager(cacheManager).build();
161  * </pre>
162  *
163  * <p><b>NOTE</b>: it should be noted that <a href="http://memcached.org/">Memcache</a> <em>DOES NOT</em> guarantee cache
164  * coherency.  It is strongly recommended that you do not use Memcache as your clustered caching solution (memcache
165  * is fine for caching files, etc, but not data that is expected to be coherent across multiple cluster nodes).</p>
166  *
167  * <h4>Disable Caching</h4>
168  *
169  * <p>While production applications will usually enable a working CacheManager as described above, you might wish to disable caching
170  * entirely when testing or debugging to remove 'moving parts' for better clarity into request/response
171  * behavior.  You can do this by configuring a <em>disabled</em> {@code CacheManager} instance. For example:</p>
172  *
173  * <pre>
174  * Client client = {@link com.okta.sdk.client.Clients Clients}.builder().setCacheManager(
175  *     <b>{@code Caches.newDisabledCacheManager()}</b>
176  * ).build();
177  * </pre>
178  *
179  * <h1>Single Instance</h1>
180  *
181  * <p>Finally, it should be noted that, after building a {@code client} instance, that same instance should be used
182  * everywhere in your application. Creating multiple client instances in a single application could have
183  * negative side effects:</p>
184  *
185  * <p>As mentioned above, a client has a {@link #setCacheManager(com.okta.sdk.cache.CacheManager) CacheManager}
186  * reference.  If your application uses multiple client instances, each client's referenced CacheManager would likely
187  * become out of sync with the others, making your cache
188  * <a href="http://en.wikipedia.org/wiki/Cache_coherence">incoherent</a>.  This will likely
189  * result in exposing stale data to your application and could data errors.</p>
190  *
191  * <p>If you must have multiple {@code Client} instances in your application, you should ensure that each client
192  * references the same exact {@code CacheManager} instance to guarantee cache coherency.</p>
193  *
194  * @since 0.5.0
195  */

242     /**
243      * Sets the {@link CacheManager} that should be used to cache Okta REST resources, reducing round-trips to the
244      * Okta API server and enhancing application performance.
245      *
246      * <h1>Single JVM Applications</h1>
247      *
248      * <p>If your application runs on a single JVM-based applications, the
249      * {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder} should be sufficient for your needs. You
250      * create a {@code CacheManagerBuilder} by using the {@link com.okta.sdk.cache.Caches Caches} utility class,
251      * for example:</p>
252      *
253      * <pre>
254      * import static com.okta.sdk.cache.Caches.*;
255      *
256      * ...
257      *
258      * Client client = Clients.builder()...
259      *     .setCacheManager(
260      *         {@link com.okta.sdk.cache.Caches#newCacheManager() newCacheManager()}
261      *         .withDefaultTimeToLive(1, TimeUnit.DAYS) //general default
262      *         .withDefaultTimeToIdle(2, TimeUnit.HOURS) //general default
263      *         .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(User.class) //User-specific cache settings
264      *             .withTimeToLive(1, TimeUnit.HOURS)
265      *             .withTimeToIdle(30, TimeUnit.MINUTES))
266      *         .withCache({@link com.okta.sdk.cache.Caches#forResource(Class) forResource}(Group.class) //Group-specific cache settings
267      *             .withTimeToLive(2, TimeUnit.HOURS))
268      *         .build() //build the CacheManager
269      *     )
270      *     .build(); //build the Client
271      * </pre>
272      *
273      * <p><em>The above TTL and TTI times are just examples showing API usage - the times themselves are not
274      * recommendations.  Choose TTL and TTI times based on your application requirements.</em></p>
275      *
276      * <h1>Multi-JVM / Clustered Applications</h1>
277      *
278      * <p>The default {@code CacheManager} instances returned by the
279      * {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder} might not be sufficient for a
280      * multi-instance application that runs on multiple JVMs and/or hosts/servers, as there could be cache-coherency
281      * problems across the JVMs.  See the {@link com.okta.sdk.cache.CacheManagerBuilder CacheManagerBuilder}
282      * JavaDoc for additional information.</p>
283      *
284      * <p>In these multi-JVM environments, you will likely want to create a simple CacheManager implementation that
285      * wraps your distributed Caching API/product of choice and then plug that implementation in to the Okta SDK
286      * via this method.  Hazelcast is one known cluster-safe caching product, and the Okta SDK has out-of-the-box
287      * support for this as an extension module.  See the top-level class JavaDoc for a Hazelcast configuration
288      * example.</p>
289      *
290      * @param cacheManager the {@link CacheManager} that should be used to cache Okta REST resources, reducing
291      *                     round-trips to the Okta API server and enhancing application performance.
292      * @return the ClientBuilder instance for method chaining
293      */
294     ClientBuilder setCacheManager(CacheManager cacheManager);
